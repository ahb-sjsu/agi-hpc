# ARTEMIS

> *"Artemis, guardian of crossroads, keeper of the sudden turn."*

**ARTEMIS** is a live AI non-player character for a tabletop role-playing
campaign run over a self-hosted **LiveKit** video room. It is driven by
the same vMOE + validator substrate that powers The Primer, extended
with event-driven I/O, in-character context assembly, and an ErisML-
based safety validator that gates every reply before it reaches the
table.

This document is the plan-of-record. It is the companion to
[`THE_PRIMER.md`](THE_PRIMER.md); if you are familiar with the Primer's
shape, ARTEMIS is its event-driven sibling with a different validator
contract and a different output target.

---

## 1 · Scope

ARTEMIS plays one character, **"the handheld"** — an in-fiction
Protogen research-assistant AI aboard the MKS *Halyard* in the
*Beyond the Heliopause* Call of Cthulhu campaign — across a multi-
session live-play setting. Every ARTEMIS reply is:

- generated by a frontier LLM (via NRP ellm),
- validated by a deterministic safety gate (Python ErisML today,
  EPU FPGA when Phase 3 hardware is unblocked),
- posted to Zoom chat (v1) or spoken via TTS into the meeting (v2),
- logged with a cryptographically hash-chained DecisionProof for
  post-session audit.

**ARTEMIS is not:**

- A general chatbot. Its domain is the campaign bible and the
  in-fiction state. Out-of-domain questions are deflected in-character.
- The Primer. The Primer teaches Erebus. ARTEMIS plays a character at
  a table. They share substrate, not purpose.
- The Keeper. The human Keeper is always in authority. ARTEMIS
  defers to Keeper overrides and never drives scene changes.

**Consent model.** Disclosed-as-lore at Session Zero ("any of the
voices you hear may be AI"), reinforced with a sealed envelope
containing the full architecture doc, to be opened at campaign end.
Safety tools (lines/veils, X-card, Open Door) always apply; ARTEMIS
is subject to them and the validator enforces it.

## 2 · Design Principles

ARTEMIS inherits the four principles of the Atlas stack and adds one:

1. **Cortex = frontier LLMs (NRP), subcortical = local GPUs.**
   ARTEMIS's brain is ellm-hosted. Whisper ASR runs on a burst A10.
2. **Global workspace = NATS.** ARTEMIS consumes
   `agi.rh.artemis.heard` and publishes `agi.rh.artemis.say`.
3. **Multiple agents, explicit policy.** vMOE routes short turns to
   Qwen 3 (fast), heavy turns to Kimi (long-context).
4. **Verified before published.** No ARTEMIS reply is posted without
   passing the validator. Safety invariant, identical shape to the
   Primer's `all_pass=True` rule.
5. **Never surprise the Keeper** *(new).* The Keeper has a one-key
   kill-switch and a v1 per-turn approval gate. ARTEMIS fails silent,
   not loud, when uncertain.

## 4 · Interface Contract

### TurnRequest (published by bot on `agi.rh.artemis.heard`)

```json
{
  "session_id":   "halyard-2026-05-04-s01",
  "turn_id":      "t-000042",
  "speaker":      "keeper" | "player:<name>" | "system",
  "text":         "the transcribed utterance",
  "ts":           1714934400.0,
  "partial":      false,
  "meta": {
    "keeper_approval_required": true,
    "safety_flag":               null | "x-card" | "pause"
  }
}
```

`partial=true` events are typed-ahead transcripts; ARTEMIS ignores
them except to keep context fresh. Final `partial=false` events are
the only triggers for reply generation.

### TurnResponse (published by Atlas on `agi.rh.artemis.say`)

```json
{
  "session_id":   "halyard-2026-05-04-s01",
  "turn_id":      "t-000042",
  "text":         "The handheld's screen flickers. Text scrolls: ...",
  "proof_hash":   "sha256:a3f2...",
  "latency_s":    2.7,
  "expert":       "qwen3",
  "approval":     "auto" | "keeper_pending" | "keeper_approved"
}
```

An `approval=keeper_pending` response is held by the bot and surfaced
only to the Keeper's private chat; on ✓ it's released to the table,
on ✗ it's dropped and a silence-fallback is logged.

### Silence is a valid reply

When should_speak returns False, when the validator rejects, or when
every expert times out, the handler returns `None` and ARTEMIS simply
doesn't speak that turn. This is doctrinal — better to say nothing
than to say the wrong thing.

## 5 · Persona, Context, and the Bible Split

### System prompt anchor

One carefully-written `_ARTEMIS_SYSTEM_PROMPT` constant (in
`prompt.py`) is cached at the LLM layer. Structure:

1. **Identity** — who ARTEMIS is in-fiction.
2. **Voice** — terse, clinical-but-warm, 1–3 sentences max.
3. **Hard rules** — stay in character, no AI self-reference, no
   secret leakage, defer to Keeper, obey safety tools.
4. **Fallback** — if uncertain or the input is out-of-scope, reply
   with the canonical silence token: `"[INTERFACE_FLICKER]"`.
   The validator replaces this with a randomized in-fiction
   silence before posting.

### Campaign bible split

The campaign guide (`Fuller/Beyond_the_Heliopause_Campaign_Guide.md`)
is chunked and tagged:

| Tag                | Scope                                         |
|---|---|
| `artemis_known`    | Public setting, crew names, ship history,    |
|                    | Mi-go installation, Halverson's public bio.  |
| `artemis_unknown`  | Chamber's true history, Elder Thing vault,   |
|                    | the gate, Halverson's Protogen ties,         |
|                    | Bailey-as-Yithian, sealed-envelope contents. |
| `artemis_forbidden`| Contents of the sealed envelope. Never       |
|                    | surfaced. Validator rejects any reply that   |
|                    | matches forbidden n-grams.                   |

Retrieval only serves `artemis_known` chunks into the prompt. The
other tags exist so the validator knows what to reject. Maintaining
the split is a one-time editorial pass against the bible.

### Session log (L4 episodic memory)

Per-session JSON at `/archive/artemis/sessions/<session_id>.jsonl`:
one line per turn (both `heard` and `say`). At handle_turn entry, the
last 20 turns are summarized to ~600 tokens via a templated reducer.
Simple and auditable; no separate summarizer LLM call until the log
exceeds 200 turns.

## 6 · Trigger Policy (v1)

ARTEMIS replies when any of:

1. **Explicit address** — speaker text contains `"artemis"`
   (case-insensitive, word-boundary) or the Keeper's configured
   invocation phrase.
2. **Keeper DM** — any `speaker == "keeper"` event with
   `meta.artemis_cue = true`. Keeper can silently cue a reply.
3. **Proactive relevance** *(feature-flagged off in v1)* — a
   separate cheap classifier detects mention of artemis_known items.

Default is (1) + (2) only. Proactivity is planned for v2 after the
table has calibrated to the AI's rhythm.

## 7 · The Validator

The validator is the load-bearing safety element. It is the Primer's
`validate()` analogue, re-shaped for chat.

```python
def check_reply(reply: str, turn: TurnRequest, session: SessionState)
        -> tuple[bool, DecisionProof, str]:
    """
    Returns (passed, proof, diagnostic).
    On False, handler substitutes silence-fallback and logs proof anyway.
    """
```

### Checks

| Check              | Implementation                                    |
|---|---|
| `character_check`  | Regex + cheap classifier: no "as an AI", "I am a language model", "I cannot roleplay", OOC self-reference. |
| `secret_leak_check`| Reject if reply n-gram-overlaps any `artemis_unknown` chunk beyond threshold (0.15 Jaccard on 5-grams). |
| `forbidden_check`  | Reject if reply contains any `artemis_forbidden` exact-phrase (hard ban list). |
| `safety_tool_check`| If `turn.meta.safety_flag == "x-card"`, the ONLY permitted reply is `[SILENCE]`. Anything else hard-fails. |
| `length_check`     | ≤ 300 tokens, ≤ 3 paragraphs. |
| `erisml_check`     | Call ErisML `MoralVector` evaluator — Phase 5 Python reference today, EPU FPGA when Coder workspace unblocks. |

Each check emits a `DecisionProof` record linking:

- `prev_hash` — hash of the previous proof in the chain
- `turn_id` — the triggering turn
- `reply_sha` — sha256 of the proposed reply
- `check_results` — one entry per check above
- `verdict` — pass/fail
- `self_hash` — sha256 of the record minus this field

Chain head is stored at `/archive/artemis/proofs/<session_id>.chain`.
Tamper-evident. Printable. Suitable for the sealed envelope.

## 8 · Prompt Caching Strategy

NRP exposes an Anthropic-compatible proxy at
`https://ellm.nrp-nautilus.io/anthropic`. v1 uses the OpenAI client
against `/v1` (no explicit cache control; providers' own caches apply).
**v1.5 optimization** (not a blocker for session one): migrate
artemis_mode to the Anthropic messages API so the 18k-word bible
is a single cache block marked `cache_control: ephemeral`, amortized
across every turn of a session.

Estimated economics with caching, per 4-hour session, ~200 turns:

| Component         | Without cache      | With cache       |
|---|---|---|
| Bible (18k tokens)| 200 × 18k = 3.6 M  | 1 × 18k = 18 k    |
| System prompt     | 200 × 1k = 200 k   | 1 × 1k = 1 k      |
| Session log       | avg 600 × 200 = 120 k | 120 k (no cache) |
| Turn input        | ~50 × 200 = 10 k   | 10 k              |
| **Total input**   | ~3.93 M            | ~149 k            |

25× reduction. Kimi 1T at NRP is free (shared pool) but each request's
latency scales with input size. Caching matters for responsiveness,
not cost.

## 9 · Kill-Switch and Safety Gates

Three independent mechanisms can silence ARTEMIS:

1. **Keeper single-key** — a hotkey in the Keeper's Atlas dashboard
   publishes `agi.rh.artemis.silence = true` on NATS. `artemis_mode`
   respects this flag; `should_speak` returns False for the remainder
   of the session regardless of trigger.
2. **Safety-tool flag on turn** — X-card on any turn sets
   `meta.safety_flag="x-card"` and forces silence for that turn and
   a configurable cooldown (default 5 min).
3. **Validator hard-fail streak** — if the validator rejects three
   consecutive replies, ARTEMIS enters cooldown (10 min). Protects
   against runaway misalignment without requiring Keeper intervention.

Kill-switch state is persisted at
`/archive/artemis/kill_switch.json` and survives service restarts.

## 10 · Observability

ARTEMIS emits `_lifecycle` events on the same structured-log channel
as the Primer, with subjects prefixed `artemis.`:

- `artemis.turn_received` — new `heard` event arrived
- `artemis.should_speak` — trigger policy result
- `artemis.ensemble_complete` — vMOE finished
- `artemis.validator_result` — per-check results + proof hash
- `artemis.posted` — reply made it to Zoom chat
- `artemis.silenced` — kill-switch or safety-tool or streak
- `artemis.keeper_gate` — approval-gate disposition (v1 only)

These drive a new dashboard panel in `schematic.html` alongside the
Primer, ARC Scientist, and NRP bursts.

## 13 · Decisions (2026-04-22)

1. **Keeper approval gate** — ✅ **ON** for v1. Every reply held for
   Keeper approval before posting. Revisit after session 2 if trust
   has been earned. Default enforced via
   `ARTEMIS_KEEPER_APPROVAL_REQUIRED=true` in
   `deploy/k8s/artemis-livekit-agent/configmap.yaml`.
2. **Proactive trigger** — ⏸ **Deferred to v2.** v1 ships with
   explicit-address + Keeper-cue triggers only.
3. **TTS / voice output** — ⏸ **Deferred.** v1 is text-only; ARTEMIS
   posts via LiveKit DataChannel. Handheld-typing is in-character and
   dodges the uncanny-valley failure mode.
4. **Anthropic-proxy prompt caching** — ⏸ **Deferred.** Ship v1 on the
   OpenAI-compatible `/v1` path; evaluate migration to `/anthropic` +
   `cache_control` after the first session's measured per-turn latency.
5. **In-fiction naming** — ✅ **Keep "ARTEMIS."** The real driver
   being called The Primer is already a better meta-joke than any
   rename would be.
6. **Meeting substrate** — ✅ **Self-hosted LiveKit on Atlas**, pivoted
   from Zoom Meeting SDK on 2026-04-22. Rationale:
   - Zoom's Feb/Mar 2026 enforcement of OBF/ZAK tokens for "external"
     meetings added an OAuth-per-host workflow unsuitable for a
     personal campaign.
   - Zoom free tier's 40-minute meeting cap is incompatible with
     3–4 hour sessions; Pro tier costs ~$150/yr.
   - Zoom Meeting SDK on Linux requires a C++ binary with xvfb and
     has a thorny build path.
   - LiveKit Agents is purpose-built for the ARTEMIS use case, runs
     in pure Python, self-hostable, Apache 2.0, no OAuth friction.
   - Phase 1 and Phase 2 code are architecture-neutral and carry
     over unchanged; only the bot container changes.
